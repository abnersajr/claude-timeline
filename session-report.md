## 1. Context of the Problem

### 1.1 Primer: Core Claude Code Concepts (Technical Team Reference)
This section defines key terms used throughout the report for alignment across the team:
| Term | Definition | Example |
|------|------------|---------|
| **Session** | A single Claude Code interaction lifecycle, started when launching `claude` or running a slash command. All work shares a persistent conversation history. | A 30-minute session to fix a bug, with 15 back-and-forth turns. |
| **Turn** | A single API call to the Claude model. Each turn includes cached context, new input, and model output. | The model responding to your prompt, or requesting to run a `Bash` command. |
| **Token** | A unit of text (~4 English characters = 1 token). All billing is token-based. | "Hello world" = ~3 tokens. |
| **Prompt Caching** | Claude feature to cache repeated context (system prompt, conversation history) to avoid re-processing. Cached context is billed at 10% of standard input rates. | System prompt cached once, reused for all 28 turns of a session. |
| **Cache Creation** | One-time cost to write context to the cache (billed at standard input rates: $3.00/MTok for Sonnet 4). | Initial 34k tokens written to cache at session start: ~$0.10. |
| **Cache Read** | Per-turn cost to retrieve cached context (billed at 10% of standard input: $0.30/MTok for Sonnet 4). | Reading 34k cached tokens per turn: ~$0.01 per turn. |
| **Billing Token Types** | - Input: New text sent to the model (not cached) <br> - Output: Text/tool requests generated by the model ($15.00/MTok for Sonnet 4) <br> - Cache Creation: One-time cache write cost <br> - Cache Read: Per-turn cache retrieval cost | 14k output tokens = ~$0.21. |
| **Tool** | Pre-defined functions Claude can invoke to interact with the local environment, filesystem, or user. Examples include `Bash` (terminal commands), `Read` (file content reads), `Edit` (file modifications), `Write` (file creation), and `AskUserQuestion` (user input prompts). | `Bash` executing `npm install`; `Read` loading `src/index.ts`. |
| **Tool Call** | A single execution request for a Tool: Claude generates a tool request, the tool runs, and its result (success output or error message) is returned. All Tool Calls and their results are appended to the session's conversation history. | Claude requests `Read` on `missing-file.txt` → Tool returns `File not found` error → Error is added to conversation history. |

#### 1.1.1 How Tools & Tool Calls Impact Turns
- A single Turn (Claude model API call) may include requests for one or more Tool Calls.
- Tool execution and results do not create new Turns: the Tool result is added to the conversation history, and the next Turn's input includes this full updated history.
- Failed Tool Calls (e.g., invalid command, missing file) are still added to conversation history, just like successful calls.
- Example Turn flow:
  1. Turn N: Claude requests `Bash` command `ls -la` → Tool executes, returns directory listing.
  2. Turn N+1: Claude's input includes the full conversation history up to Turn N, including the `ls -la` result.

#### 1.1.2 How Tools & Tool Calls Impact Billing
- Tool results (success or failure) are appended to the conversation history, which is cached after each Turn via Cache Creation tokens.
- Every subsequent Turn reads the entire cached history (including all past Tool results) via Cache Read tokens, billed at 10% of standard input rates.
- Large Tool results (e.g., reading a 50k-token log file, running a command with 10k lines of output) bloat the conversation history permanently: every future Turn's Cache Read will include this large result, compounding costs.
- Failed Tool Calls still incur billing impact: even a short error message is added to history, and the subsequent Turn's Cache Read will include it.
- Example: Reading a 100k-token file in Turn 5 will add 100k tokens to the conversation history. Turns 6–28 will each read an additional 100k cached tokens, adding ~$0.03 per turn ($0.30/MTok * 0.1MTok) for 23 turns = ~$0.69 extra cost.

### 1.2 The Problem Statement
On May 7, 2026, a team member reported an unusually high cache read token count of **929,057 tokens** in a single Claude Code session used to set up the `claude-hud` plugin.

Initial confusion arose because:
1. The 929k number sounded alarmingly high (equivalent to ~232 pages of English text)
2. It was unclear if this was a billing error, a plugin bug, or expected behavior
3. The team had no framework to audit token usage for future sessions
4. Unclear how working directory scope (e.g., starting a session in `~/` vs `~/projects/busbud/pubweb`) impacts context bloat via unnecessary tool discovery

This report documents the full investigation, turn-by-turn breakdown, and actionable workflow improvements to avoid unnecessary token bloat. It also addresses a critical team question: how working directory scope impacts context size and token usage.

## 2. Pricing, Tokens, Caching, and Billing Explained

This section breaks down the exact billing mechanics using data from the example session (`19500eaa-3cc6-4111-a82d-f158e7f76ad3`) and small hypothetical examples to illustrate edge cases.

*Note: Cache Read is 10% of standard Input rates, Cache Creation is 125% of standard Input rates (Anthropic prompt caching pricing).*

### 2.2 Token Types (With Actual Session Data)
Every Claude Code interaction uses 4 billable token types, all tracked in the `usage.db` SQLite database:
1. **Input Tokens**: New text sent to the model that is *not* cached. In the example session, only 30 input tokens were used (almost all context was cached).
   - *Small Example*: Sending a 100-token prompt with no cached context = 100 input tokens ($0.0003).
2. **Output Tokens**: Text, tool requests, or thinking generated by the model. This is the most expensive token type.
   - *Actual Session*: 14,550 output tokens (~$0.22), mostly model responses and tool request generation.
   - *Small Example*: Model generating a 500-token explanation = 500 output tokens ($0.0075).
3. **Cache Creation Tokens**: One-time cost to write context (system prompt, conversation history) to the cache. Only charged when new content is added to the cache.
   - *Actual Session*: 34,383 cache creation tokens (~$0.13) to write the initial system prompt + tools to cache.
   - *Small Example*: Caching a 10k-token system prompt = 10k cache creation tokens ($0.0375).
4. **Cache Read Tokens**: Per-turn cost to retrieve cached context. Charged every time the model reads the cached prompt.
   - *Actual Session*: 929,057 cache read tokens (~$0.28) across 28 turns.
   - *Small Example*: Reading a 10k-token cached prompt for 5 turns = 50k cache read tokens ($0.015).

### 2.3 How Prompt Caching Works (With Session Data)
Prompt caching avoids re-processing repeated context (system prompt, tools, conversation history) by storing it in a cache. Here's the lifecycle using the example session:
1. **Turn 1 (Initial Cache Write)**:
   - Model receives: System prompt (skills, AGENTS.md, tool definitions) + user command `/claude-hud:setup`
   - Cache Creation: 12,973 tokens (writes initial context to cache)
   - Cache Read: 12,143 tokens (reads the just-written cache for this first turn)
   - *Why?* The first turn always writes and reads the initial cache.
2. **Turn 2–28 (Cache Reuse)**:
   - Model receives: Cached context (system + all previous conversation) + new tool results
   - Cache Creation: ~410 tokens per turn (adds previous turn's output to the cache)
   - Cache Read: Grows from 25k (Turn 2) to 46k (Turn 28) as conversation history expands
   - *Why?* Each turn adds ~1–3k tokens of new conversation history, so the cached context grows steadily.

#### 2.3.1 How Tools/Tool Calls Impact Caching
- All Tool Calls (e.g., `Bash`, `Read`) and their results (success/failure) are added to the conversation history.
- Failed Tool Calls (e.g., `InputValidationError` found in the session) still add to history, bloating the cache.
- *Actual Session Example*: A failed `Bash` command with a 500-token error message adds 500 tokens to the conversation history. All 27 subsequent turns will include this 500 tokens in their cache read (~$0.0015 extra per turn).
- *Small Example*: Reading a 100k-token log file in Turn 5 adds 100k tokens to history. Turns 6–28 (23 turns) will each read an extra 100k cached tokens: 23 * 100k * $0.30/MTok = ~$0.69 extra cost.

### 2.4 Billing Calculation (Exact Session Math)
The example session's total cost is derived from the `sessions` table in `usage.db`:
| Component | Calculation | Cost |
|-----------|--------------|------|
| Cache Creation | 34,383 tokens / 1M * $3.75 | ~$0.13 |
| Cache Read | 929,057 tokens / 1M * $0.30 | ~$0.28 |
| Input (non-cached) | 30 tokens / 1M * $3.00 | ~$0.00 |
| Output | 14,550 tokens / 1M * $15.00 | ~$0.22 |
| **Total** | | **~$0.63** |

*Key Takeaway*: The 929k cache read number sounds large, but at $0.30/MTok it only costs ~$0.28. The most expensive component is output tokens (~$0.22), even though they are only 1.5% of the total token count.

### 2.5 Small Example: 1-Turn vs 28-Turn Session
To illustrate how cache read compounds over turns:
| Session Type | Turns | Cache Creation | Cache Read | Total Cost |
|--------------|-------|----------------|------------|------------|
| 1-Turn (quick question) | 1 | 10k ($0.0375) | 10k ($0.003) | ~$0.04 |

*Why the 28-turn session only costs ~$0.63*: Cache read is 10x cheaper than input, so even 929k cache read tokens cost less than 100k input tokens.

## 3. Investigation Methodology

This section documents the systematic approach used to identify the root cause of the 929,057 cache read tokens reported in the example session. The investigation followed a structured diagnostic workflow to isolate variables and validate findings.

### 3.1 Initial Problem Triage
When the 929k cache read report was received, the team formulated three initial hypotheses:
1. **Billing Error**: Anthropic's token counting system misreported cache read usage.
2. **Plugin Bloat**: The `claude-hud` plugin (19KB setup command, 36MB total install) was loading excessive context into the cache.
3. **Session Normalcy**: The 929k is a cumulative total across 28 turns, not a per-turn anomaly.

### 3.2 Investigation Tools & Data Sources
The following tools and data sources were used to audit the session:
| Tool/Source | Purpose | Example Usage |
|-------------|---------|---------------|
| `~/.claude/usage.db` (SQLite) | Primary token accounting database tracking sessions, turns, and token types | `sqlite3 usage.db "SELECT * FROM sessions;"` |
| `~/.claude/projects/*/session.jsonl` | Raw session transcript with tool calls, results, and model messages | `jq` parsing of JSONL files |
| `bash` + `du`/`ls` | Filesystem analysis of plugin sizes, cache directories, and config files | `du -sh ~/.claude/plugins/claude-hud/` |
| `jq` | JSON/JSONL parsing to extract tool calls, errors, and conversation flow | Extracting `Read` tool file paths from session transcripts |

### 3.3 Step-by-Step Investigation Flow

#### 3.3.1 Locate Token Usage Storage
- **Action**: Explored `~/.claude` directory structure to identify where token data is stored.
- **Finding**: Discovered `usage.db` (52KB SQLite database) containing all session/turn token accounting.
- **Key Command**: `sqlite3 ~/.claude/usage.db ".tables"` → Revealed `sessions`, `turns`, `processed_files` tables.

#### 3.3.2 Query Sessions by Cache Read Volume
- **Action**: Queried `sessions` table to find the session with the highest `total_cache_read`.
- **Finding**: Session `19500eaa-3cc6-4111-a82d-f158e7f76ad3` had exactly 929,057 cache read tokens (matched user report).
- **Key Query**:
  ```sql
  SELECT session_id, total_cache_read, total_input_tokens, turn_count 
  FROM sessions 
  ORDER BY total_cache_read DESC LIMIT1;
  ```

#### 3.3.3 Validate Turn-Level Token Patterns
- **Action**: Queried `turns` table for the identified session to see per-turn cache behavior.
- **Finding**: 28 turns total; cache read grew from 12,143 (Turn 1) to 46,119 (Turn 28); every turn had cache creation tokens (new content added to cache).
- **Key Query**:
  ```sql
  SELECT timestamp, cache_read_tokens, cache_creation_tokens, tool_name 
  FROM turns 
  WHERE session_id='19500eaa-3cc6-4111-a82d-f158e7f76ad3' 
  ORDER BY timestamp;
  ```

#### 3.3.4 Cross-Reference Session Transcript
- **Action**: Examined the session's JSONL file (`19500eaa-3cc6-4111-a82d-f158e7f76ad3.jsonl`) to map turns to actual user/assistant messages and tool calls.
- **Finding**: Session ran `/claude-hud:setup` command; included a failed `Bash` tool call with `InputValidationError`; 28 turns matched the `turns` table count.
- **Key Insight**: Failed tool calls (like the `InputValidationError`) still add to conversation history, bloating future cache reads.

#### 3.3.5 Audit Plugin & Context Size
- **Action**: Checked `claude-hud` plugin installation size and contents to see if large files were loaded into context/cache.
- **Finding**: 
  - Total plugin install size: 36MB (34MB in `node_modules/`, 2MB in source/docs, 19KB in `commands/setup.md`).
  - **Only the `commands/setup.md` file (19KB, ~4.7k tokens) was loaded into the session context and cached** — the 34MB `node_modules/` directory and other large files (README, CHANGELOG) were never read by the model.
  - The 34,383 cache creation tokens (~34k tokens) align with: System prompt (~25k tokens) + `setup.md` (~4.7k tokens) + tool definitions (~4k tokens) = ~33.7k tokens (matches the 34k cache creation total).
- **Key Commands**:
  ```bash
  # Check total plugin size
  du -sh ~/.claude/plugins/cache/claude-hud/claude-hud/0.1.0/  # Returns 36MB
  
  # List only files loaded into context (commands)
  ls -la ~/.claude/plugins/cache/claude-hud/claude-hud/0.1.0/commands/  # setup.md (19KB), configure.md (12KB)
  
  # Verify node_modules was not accessed (no Read tool calls for it)
  jq -r 'select(.message[0].name=="Read") | .message[0].input.filePath' session.jsonl  # No node_modules paths found
  ```
- **Critical Takeaway**: Claude Code only loads explicitly referenced plugin files (e.g., the command you invoke) into context — it does not recursively read entire plugin directories. The 36MB total size is irrelevant to token usage; only the 19KB `setup.md` contributed to cache.

#### 3.3.6 Calculate Actual Billing Impact
- **Action**: Used session token totals + Anthropic Sonnet 4 pricing to compute real cost.
- **Finding**: 929k cache read = ~$0.28; total session cost ~$0.63 (mostly output tokens: ~$0.22).
- **Validation**: Confirmed cache read rate is 10% of input rate ($0.30/MTok vs $3.00/MTok).

### 3.4 Investigation Outcome
The methodology confirmed **Hypothesis 3 (Session Normalcy)**:
- 929,057 cache read tokens = sum across 28 turns (average ~33k per turn)
- Cache read grows because each turn adds conversation history (including tool results) to the cached context
- No billing error; no plugin bloat; expected behavior for a 28-turn session with cumulative cache reads

### 4.3 Working Directory Scope & Context Bloat (Critical Team Question)
You asked: *If I work on `~/projects/busbud/pubweb/worktrees/worktree-a`, start a session in `~/` or `~/projects/busbud/pubweb`, even if I give the exact file, does it make a difference for discovery? Can a broader scope (like `~/`) add trash to context?*

#### 4.3.1 Direct Answer
**Yes, working directory scope makes a significant difference for discovery and context bloat**, even if you provide exact file paths. Here's why:

| Scenario | Working Directory | Discovery Behavior | Context Bloat Risk |
|----------|-------------------|-------------------|--------------------|
| **Project-Specific Work** | `~/projects/busbud/pubweb` | Discovery tools (Glob, Grep, Read) scope to this directory. Finds only project-relevant files. | Low: Only project files are discovered. |
| **Root Directory Work** | `~/` | Discovery tools start at `~`, searching all home files (Documents, Downloads, other projects). | High: Unrelated files (e.g., old logs, other repos) may be accidentally read and added to cache. |
| **Exact File Path Given** | Either | Claude uses the exact path directly, no discovery needed. | Minimal: No discovery = no unrelated files. But if Claude later uses discovery (e.g., "find all tests for this component"), broader scope adds trash. |

#### 4.3.2 Example: Component X in `worktree-a`
Suppose you ask Claude to "fix component X" in `worktree-a`:
- **Session started in `~/`**: Claude runs `Glob` for "component X" starting at `~`. It finds files in `worktree-a`, but also in `~/Downloads/old-component-x`, `~/projects/other-repo/component-x`. If it reads these unrelated files, they bloat the cache permanently.
- **Session started in `~/projects/busbud/pubweb`**: Claude's `Glob` scopes to `pubweb`, finds only relevant files. No trash added.

#### 4.3.3 Impact on Cache & Billing (From Example Session)
In the example session (working directory `~/`):
- The task was user-level plugin setup, so no project discovery was needed. No trash was added.
- For project work: 1 unrelated 10k-token file read (due to broad discovery) adds 10k tokens to history. For a 28-turn session, this adds 10k * 28 = 280k extra cache read tokens (~$0.08 extra cost) + 10k cache creation (~$0.04). Total extra: ~$0.12 per unrelated file.

#### 4.3.4 Prompt Strategies to Prevent Context Bloat
1. **Always start sessions in the project root** you're working on (e.g., `cd ~/projects/busbud/pubweb && claude`).
2. **Use exact file paths** instead of letting Claude discover broadly: "Edit `src/components/X.tsx`" instead of "Edit component X".
3. **Use `.claudeignore`** to exclude unnecessary dirs (e.g., `node_modules/`, `dist/`, `old-logs/`) from discovery.
4. **Limit discovery scope explicitly**: "Search for X only in `src/components/`".
5. **Avoid broad commands** like "find all files using Y" without scoping.
6. **Monitor tool outputs**: If a `Glob` or `Grep` returns too many results, cancel and narrow the scope before Claude reads them all.

## 5. Conclusion & Workflow Improvements

### 5.1 Summary of Findings
The investigation into the reported 929,057 cache read tokens in session `19500eaa-3cc6-4111-a82d-f158e7f76ad3` revealed the following:

1. **The 929k is cumulative, not per-turn**: The session had 28 turns, with an average cache read of ~33k tokens per turn. The total (929k) is the sum of all 28 turns' cache reads, not a single turn anomaly.

2. **Cache read growth is expected behavior**: Each turn adds conversation history (tool calls, results, user/assistant messages) to the cached context. Turn 1 read 12k tokens; Turn 28 read 46k tokens. This linear growth is normal for extended sessions.

3. **Actual billing impact is minimal**: Despite the large 929k number, cache read tokens are billed at 10% of input rates ($0.30/MTok). The total session cost was ~$0.63, with output tokens (~$0.22) being the most expensive component.

4. **Working directory scope impacts context bloat**: Starting sessions in broader directories (e.g., `~/`) increases the risk of unrelated files being discovered and added to cache. The example session avoided this by not requiring project discovery.

5. **Tool call results permanently affect cache**: All tool outputs (success/failure) are added to conversation history. Failed tool calls (like the `InputValidationError` found in Turn 8) still bloat future cache reads.

### 5.2 Why Cache Read Numbers Grow (The Mechanics)
Cache read tokens grow because of how prompt caching works:
- **Cache Creation (Turn N)**: Adds Turn N-1's output to the cached context (e.g., 410 tokens in Turn 2 for Turn 1's output).
- **Cache Read (Turn N)**: Reads the entire cached context (system prompt + all history up to Turn N-1).
- **The Math**: If cached context = 34k tokens, Turn N's cache read = 34k. Turn N+1's cache read = 34k + Turn N's output. Over 28 turns, this compounds to 929k total.

**Example from Session**:
- Turn 11 read `setup.md` (~4.7k tokens) → Added to history
- Turns 12–28 each read an extra 4.7k tokens → 17 turns × 4.7k = 79.9k extra cache reads
- Cost of this one file: 79.9k × $0.30/MTok = ~$0.02

### 5.3 How to Reduce Turns (Shorter Sessions = Lower Cumulative Cache Read)
The most effective way to reduce total cache read tokens is to reduce the number of turns:

| Strategy | Impact | Example |
|----------|--------|---------|
| **Batch related tasks** | Fewer turns = less cumulative cache read | "Fix X, Y, Z" in one session vs three separate sessions |
| **Use specific prompts** | Reduces back-and-forth clarification turns | "Edit `src/X.tsx` line 42 to fix null check" vs "Fix X" |
| **Avoid excessive AskUserQuestion** | Each question = 1 turn | Pre-configure defaults instead of asking 5 questions |
| **Limit session length** | 10 turns = ~330k cache read vs 28 turns = ~930k | Break work into 10-turn chunks |

**Math**: A 10-turn session with 34k cached context = 10 × 34k = 340k cache read (~$0.10). A 28-turn session = 28 × ~33k = 924k (~$0.28). **18 fewer turns = ~$0.18 savings.**

### 5.4 Workflow Improvements to Minimize Token Bloat

#### 5.4.1 Always Scope Your Working Directory
**Problem**: Starting in `~/` risks discovering unrelated files (old logs, other repos) that bloat cache.
**Solution**: Start sessions in the project root:
```bash
# Bad: Broad scope
cd ~/
claude "Fix component X in pubweb"

# Good: Scoped to project
cd ~/projects/busbud/pubweb
claude "Fix component X"
```

#### 5.4.2 Use Exact File Paths, Not Discovery
**Problem**: Letting Claude discover files via `Glob`/`Grep` may return unrelated matches in broader scopes.
**Solution**: Provide exact paths:
```markdown
# Bad: Broad discovery
"Edit the component X file"

# Good: Exact path
"Edit ~/projects/busbud/pubweb/src/components/X.tsx"
```

#### 5.4.3 Watch Tool Output Sizes
**Problem**: Large tool outputs (e.g., reading 100k-token log files, `ls -la` on 10k files) permanently bloat cache.
**Solution**: 
- Limit output with `head`/`tail`: `cat large-file.log | head -50`
- Use targeted searches: `grep "ERROR" app.log` vs `cat app.log`
- Ask Claude to "summarize output briefly" before reading full results

#### 5.4.4 Use `.claudeignore` to Exclude Trash
**Problem**: Discovery tools may return `node_modules/`, `dist/`, `old-logs/` files.
**Solution**: Create `.claudeignore` in project root:
```
node_modules/
dist/
build/
*.log
old/
.DS_Store
```

#### 5.4.5 Minimize Failed Tool Calls
**Problem**: Failed tool calls (invalid commands, missing files) still add error messages to history.
**Solution**:
- Verify paths exist before asking Claude to read them
- Use correct tool syntax (check tool definitions in system prompt)
- If a tool fails, fix the root cause before retrying

#### 5.4.6 Break Long Sessions into Chunks
**Problem**: 28-turn sessions compound cache read costs.
**Solution**: After 10–15 turns, restart the session to reset cache accumulation:
```bash
# After completing a chunk of work
/exit
claude  # Restart fresh, cache starts at ~12k again
```

### 5.5 Team Best Practices (General)
1. **Start sessions in project directories**, not `~/` or `/`.
2. **Provide exact file paths** instead of letting Claude discover broadly.
3. **Use `.claudeignore`** to exclude unnecessary directories from discovery.
4. **Limit tool output sizes** (use `head`, `grep`, targeted queries).
5. **Keep sessions under 15 turns** to minimize cumulative cache read.
6. **Batch related tasks** into single sessions instead of multiple short ones.
7. **Monitor token usage** via `usage.db` queries to identify bloat patterns.
8. **Prefer writing code over asking questions** (reduces AskUserQuestion turns).

### 5.6 Key Takeaways for the Team
- **929k cache read sounds scary, but costs ~$0.28** (10% of input rates).
- **Output tokens are 5× more expensive** than input ($15/MTok vs $3/MTok) — focus on reducing output, not cache read.
- **Working directory scope directly impacts context bloat** — always scope to your project.
- **Every tool result (success/failure) permanently affects cache** — watch what you ask Claude to read/run.
- **Shorter sessions = lower cumulative cache read** — break work into 10–15 turn chunks.
